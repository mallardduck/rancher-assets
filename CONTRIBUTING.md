# Contributing to rancher-assets

This guide covers the development workflow and how to contribute changes.

For architecture and design context, see [BACKGROUND.md](BACKGROUND.md).

## Prerequisites

- Go 1.25+
- Docker with buildx (for multi-arch builds)
- yq (for Makefile)
- git

## Repository Structure

```
rancher-assets/
├── config.yaml              # Static configuration (chart majors, upstream branches)
├── lock.yaml                # Dynamic state (upstream commits, generated by make generate)
├── Makefile                 # Build automation
├── main.go                  # Generator CLI
├── package/
│   └── copy-charts.sh      # Init container script (copied into images)
├── internal/
│   ├── config/             # Config loading & validation
│   ├── lockfile/           # Lock file management & diff detection
│   └── generator/          # Dockerfile template rendering
└── dockerfiles/            # Generated Dockerfiles (committed to git)
    ├── Dockerfile.v0
    └── Dockerfile.v1
```

## Configuration Files

### config.yaml (Static - Human Edited)

Defines chart major versions and their upstream branches.

**When to modify:**
- Adding new Rancher minor version (e.g., v2 for Rancher 2.16)
- Updating base image versions
- Changing upstream repository URLs

**Example:**
```yaml
chart-versions:
  v1:
    rancher-branch: release/v2.15
    prod:
      charts-branch: release-v2.15
      partner-branch: main
      rke2-branch: main
    dev:
      charts-branch: dev-v2.15
      partner-branch: main
      rke2-branch: main
```

### lock.yaml (Dynamic - Auto-Generated)

Tracks upstream commit references and script hash. Generated by `make generate`.

**When updated:**
- `make generate` - Queries upstreams, updates commit SHAs, computes script hash
- Never edit manually

**Why it matters:**
The commit SHAs in lock.yaml are baked into Dockerfiles during generation. This ensures reproducible builds: checking out the same git tag always produces identical images, regardless of when the build occurs.

## Build Types

- **Production builds** (`v1.0.0`) - Use `upstream-refs.prod` from lock.yaml
- **Development builds** (`v1.0.0-rc.1`) - Use `upstream-refs.dev` from lock.yaml

See [BACKGROUND.md](BACKGROUND.md) for how build type detection works.

## Development Workflow

### 1. Make Changes

**Updating base image:**
```bash
# Edit config.yaml
vim config.yaml  # Change base-image.bci-version
```

**Updating upstream branches:**
```bash
# Edit config.yaml
vim config.yaml  # Change prod/dev branch names
```

**Updating init container script:**
```bash
# Edit the copy script
vim package/copy-charts.sh
```

### 2. Regenerate Dockerfiles

```bash
make generate
```

This will:
1. Load `config.yaml`
2. Query upstream repos for latest commits on configured branches
3. Compute hash of `package/copy-charts.sh`
4. Generate `dockerfiles/Dockerfile.v0`, `dockerfiles/Dockerfile.v1`, etc.
5. Update `lock.yaml` with upstream commits and script hash

### 3. Review Changes

```bash
git diff dockerfiles/ lock.yaml
```

**What to check:**
- Dockerfiles reflect your config changes
- lock.yaml has updated upstream commits
- No unexpected changes

### 4. Test Locally

**Build a specific chart major:**
```bash
make build CHART_MAJOR=v1 VERSION=v1.0.0-rc.1
```

**Build all chart majors:**
```bash
make build-all
# Generates dev versions like v0.0.0-dev.20260605.abc1234
```

**Test the init container:**
```bash
docker run --rm -v /tmp/charts-test:/charts \
  rancher/rancher-charts:v1.0.0-rc.1

# Verify charts copied
ls /tmp/charts-test/
```

### 5. Verify Generated Files

```bash
make verify
```

Ensures all generated files are committed (fails in CI if you forget to run `make generate`).

### 6. Commit Changes

```bash
git add config.yaml dockerfiles/ lock.yaml package/
git commit -m "Bump BCI version to 16.1"
```

**Important:** Always commit generated files (dockerfiles/, lock.yaml) along with your changes.

## Testing Changes

### Local Build Testing

```bash
# Build dev version for v1
make build CHART_MAJOR=v1 VERSION=v1.0.0-rc.1

# Build prod version for v1
make build CHART_MAJOR=v1 VERSION=v1.0.0

# Build all majors
make build-all
```

### Init Container Testing

```bash
# Run the image to copy charts
docker run --rm -v /tmp/charts-test:/charts \
  rancher/rancher-charts:v1.0.0-rc.1

# Verify charts are present
ls -lah /tmp/charts-test/rancher-charts/
ls -lah /tmp/charts-test/rancher-partner-charts/
ls -lah /tmp/charts-test/rancher-rke2-charts/

# Check metadata
docker inspect rancher/rancher-charts:v1.0.0-rc.1 | jq '.[0].Config.Labels'
```

### Change Detection Testing

Test that lock.yaml diffs work correctly:

```bash
# Make changes to config that affect upstream refs
vim config.yaml  # Change a branch

# Regenerate
make generate

# Test change detection
git add -A
git commit -m "test: change upstream refs"

# Detect what changed
go run main.go changed-majors --from=HEAD^ --to=HEAD --verbose
# Should output: ["v1"] or whichever majors changed
```

## Fork-Friendly Configuration

The Makefile supports configurable registry/org/repo for forks:

```bash
make push-all \
  REGISTRY=ghcr.io \
  ORG=myorg \
  REPO=my-charts \
  SOURCE_REPO=myorg/rancher-assets
```

See [BACKGROUND.md](BACKGROUND.md) for default values and fork design details.

## Common Tasks

### Adding a New Chart Major (New Rancher Minor)

When Rancher 2.16 releases:

1. **Update config.yaml:**
   ```yaml
   chart-versions:
     v2:  # New chart major
       rancher-branch: release/v2.16
       prod:
         charts-branch: release-v2.16
         partner-branch: main
         rke2-branch: main
       dev:
         charts-branch: dev-v2.16
         partner-branch: main
         rke2-branch: main
   ```

2. **Regenerate:**
   ```bash
   make generate
   # Creates dockerfiles/Dockerfile.v2
   ```

3. **Update versions branch:**
   ```bash
   git checkout versions
   
   # Add v2 entry
   yq eval '.v2.stable.tag = null' -i versions.yaml
   yq eval '.v2.prerelease.tag = null' -i versions.yaml
   
   git add versions.yaml
   git commit -m "Initialize v2 for Rancher 2.16"
   git push origin versions
   
   git checkout main
   ```

4. **Commit and PR:**
   ```bash
   git add config.yaml dockerfiles/ lock.yaml
   git commit -m "Add v2 chart major for Rancher 2.16"
   ```

### Updating Base Image

```bash
# Edit config.yaml
vim config.yaml  # Change base-image.bci-version: "16.1"

# Regenerate
make generate

# Commit
git add config.yaml dockerfiles/ lock.yaml
git commit -m "Bump BCI to 16.1"
```

### Updating Upstream Branches

```bash
# Edit config.yaml
vim config.yaml  # Change prod.charts-branch or dev.charts-branch

# Regenerate
make generate

# Commit
git add config.yaml dockerfiles/ lock.yaml
git commit -m "Update v1 to track dev-v2.15 branch"
```

## Release Process

Contributors don't manage releases directly. See [RELEASE-TEAM-GUIDE.md](RELEASE-TEAM-GUIDE.md) for release operations.

**Automatic pre-releases** happen on merge to main when lock.yaml changes.

**Manual releases** are triggered by the Release Team via workflow_dispatch.

## Pull Request Guidelines

1. **Always run `make generate`** before committing
2. **Commit generated files** (dockerfiles/, lock.yaml)
3. **Run `make verify`** to ensure nothing is missing
4. **Test locally** with `make build` or `make build-all`
5. **Keep PRs focused** - one logical change per PR
6. **Update docs** if changing behavior or adding features
